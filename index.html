Heralds your passwords — one link, one read, never stored.

SealKeeper is an ephemeral, zero-knowledge password delivery service. The encryption key lives in the URL fragment, which browsers never send. The server holds an opaque ciphertext, purges it on first read, and cannot — by construction — read what it transmits.

Why

A narrow tool for a recurring problem.

Sharing a credential is a small act that organisations perform constantly and seldom well. SealKeeper sits between two unsatisfactory extremes.

I.

The wrong channels

Plain-text email, Slack threads, SMS, sticky notes under a keyboard. They leave a permanent record on devices, indexers, and backups that long outlive the need to know.

II.

The oversized alternatives

A full-blown password manager is the right answer for long-lived shared secrets. For a one-off — a new contractor, an onboarding credential, a recovery token — it is procurement, training, seats.

III.

The middle ground

One ephemeral channel: a single link, single read, with an audit-trail your DPO can keep, deployed on infrastructure you already control. Nothing more, nothing less.

How it works

Eight steps. The key never reaches us.

Encryption happens in the browser. The fragment of the URL — everything after the # — is held by the client and is never transmitted to the server by any compliant browser. The server stores a ciphertext and forgets it on the first successful read.

  1. Step 01

    Request

    An operator selects a policy — length, alphabet, lifetime — and the recipient's email.

  2. Step 02

    Generation

    A password is generated on the SealKeeper instance using crypto/rand, matching the policy exactly.

  3. Step 03

    Client-side encryption

    A fresh 256-bit key is generated in the operator's browser. The password is sealed with AES-256-GCM.

  4. Step 04

    Ciphertext upload

    Only the ciphertext, nonce, and policy metadata travel to the server. The key never leaves the browser.

  5. Step 05

    URL fragment

    The server returns an identifier. The key is appended to the URL after # — a fragment, by spec not sent in HTTP.

  6. Step 06

    Email

    The recipient receives a sober, signed message: who sent it, when it expires, the link itself, and nothing more.

  7. Step 07

    Client-side decryption

    The recipient's browser extracts the fragment, fetches the ciphertext, and decrypts locally.

  8. Step 08

    Purge

    The server marks the ciphertext consumed and purges it. A second visitor finds an audit-friendly 410 Gone.

Quickstart

Three commands. No telemetry. No external dependencies.

A SealKeeper instance is a single Go binary, distributed as an OCI image and as static builds. It speaks HTTP and SMTP and persists to SQLite or Postgres. There is nothing else to install.

I. Run the service 
# minimal local instance, persists to /var/lib/sealkeeper
docker run --rm -p 8443:8443 \
  -e SK_BASE_URL="https://seal.example.eu" \
  -e SK_SMTP_HOST="smtp.example.eu" \
  -v sealkeeper:/var/lib/sealkeeper \
  ghcr.io/sealkeeper/sealkeeper:latest
II. Mint a link 
# the response carries the URL — including its #fragment
curl -X POST https://seal.example.eu/api/v1/links \
  -H "Authorization: Bearer $SK_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "recipient": "[email protected]",
    "policy":    "strong-24",
    "ttl":       "24h"
  }'
III. Verify the seal 
# after one read, the server returns a clean 410 Gone
curl -I https://seal.example.eu/links/9f2a1b

Security

Responsible disclosure

SealKeeper is a security product. We treat reports the way we would wish ours to be treated: promptly, in writing, without legal threat. Encrypted mail is preferred; plain text is accepted. We acknowledge within seventy-two hours and aim to publish a fix and advisory within thirty days of confirmation.

An informal bug-bounty programme rewards meaningful findings with a public acknowledgement, a printed certificate of thanks, and — at our discretion — a contribution to a security non-profit of the researcher's choosing.
