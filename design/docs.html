Operating SealKeeper

From the initial Docker container to the daily generation of one-time links, this page walks you through the operational lifecycle of a SealKeeper instance.

1 · Installation

Prerequisites

  • A Linux host with Docker 24+ or Podman 4+, exposed on port 443 (or behind a reverse proxy)
  • A reachable SMTP relay (host, port, username, password) for sending recipient emails
  • A persistent volume of at least 1 GB for ciphertext storage and audit logs
  • A reverse proxy with valid TLS (nginx, Caddy, Traefik, or Cloudflare Tunnel)

Pulling and running the container

# SealKeeper runs as a single Go binary in an OCI image
docker run -d \
  --name sealkeeper \
  --restart unless-stopped \
  -p 8443:8443 \
  -e SK_BASE_URL="https://seal.example.eu" \
  -e SK_DATABASE_URL="postgres://sk:secret@db:5432/sealkeeper?sslmode=require" \
  -e SK_MASTER_SECRET="$(openssl rand -base64 32)" \
  -v sealkeeper-data:/var/lib/sealkeeper \
  ghcr.io/sched75/sealkeeper:latest
Bootstrap password. On first launch — when the admins table is empty — SealKeeper auto-creates the account admin@localhost with a random 20-character password printed once to the binary's stdout. Capture it from the container logs immediately: docker logs sealkeeper | grep "BOOTSTRAP ADMIN". The wizard will then force you to change the password and enrol a TOTP authenticator before any other admin page becomes reachable.

2 · Admin console setup

Navigate to https://your-host/admin/login. The first user is admin@localhost — its bootstrap password is printed on the binary's stdout the first time the database is migrated and never reprinted afterwards. Record it before doing anything else.

After signing in with the bootstrap password, the console forces you through the setup wizard. You have to change the password and enrol a TOTP authenticator before any other admin page becomes reachable. The wizard hands back eight recovery codes — copy them at once; they're shown exactly once.

ANSSI B3 password floor. Every admin password — bootstrap, setup wizard, later rotations on /admin/account — must reach at least 16 characters and mix three of the four classes (lower-case, upper-case, digits, symbols). The couple is equivalent to ≥ 100 bits of entropy from any typable alphabet. The validator lives in internal/admin/password.go and is hard-coded — it cannot be relaxed via configuration. PRD: FR-M.4..6.

2.1 — The dashboard

Once setup completes, the dashboard summarises the running instance: number of audit entries, current session expiry, last login timestamp. The header carries a navigation strip with links to every configuration surface.

2.2 — Renaming yourself and adding admins

The bootstrap account is named admin@localhost — useful to log in, less so for an audit trail. The /admin/account page lets the current admin rewrite their email and rotate their password. The validator on the password side is the same one the setup wizard runs; both reject anything below the ANSSI B3 floor.

Multi-operator instances are managed from /admin/admins. Any admin can create another one — the server generates a 20-character bootstrap password and reveals it once in a banner on the next page render. Hand it to the new operator out-of-band (telephone, paper); the email never carries the credential. Disable and Delete are available too, with a double-confirm input for delete and three hard guard-rails: an admin can't disable themselves, can't delete themselves, and can't reduce the count of active admins to zero (ErrLastActiveAdmin).

2.3 — Allowlisted domains

The public request form refuses to mint a link for any email whose domain isn't in the allowlist. An empty allowlist means any domain is accepted — that's the zero-config eval behaviour. As soon as you add a single entry, the gate engages. Wildcard form *.example.com matches any subdomain but not the bare apex; list both if you need both.

2.4 — Policies and the live entropy preview

Each allowed domain may carry up to three policies, one per ANSSI level (B1, B2, B3). The policy controls which generator runs (G1 citation-based, G2 Diceware, G3 random alphanumeric), how many proposals are offered, how long the reveal session lives, whether the post-consultation notification fires, and the JSON-shaped generator parameters. As you tweak the form the entropy preview re-runs in the browser using the same calculator the public reveal page uses, and tells you which ANSSI threshold each variant meets.

2.5 — Elevations (B2 / B3 lists)

By default every user of an allowed domain falls into the B1 bucket and gets the B1 policy. An elevation row promotes a specific email to B2 or B3 — typically managers and system administrators. The page accepts both single-row additions and CSV bulk import.

2.6 — Libraries (dictionaries and corpora)

The G1 (citation) and G2 (Diceware) generators each read from a library shipped with the binary. The Libraries page lets you upload your own — a dictionary file is one word per line, 4-12 letters; a corpus is one citation per line, 3-25 words. Files are stored content-addressed by their SHA-256 hash so identical uploads deduplicate.

2.7 — Email templates

The reveal-link and post-consultation emails are fully template- driven (Go's html/template + text/template syntax). Defaults ship for French and English; the page lets you override the subject + text + HTML bodies per language and per kind, preview the rendered output, and reset back to the built-in default at any time.

2.8 — Outbound integrations (SIEM)

Every audit event can be fanned out to one or more sinks: webhook, Splunk HEC, Sentinel via HMAC-SHA256, Elastic _bulk, syslog over UDP / TCP. Filters at the row level pick which event types reach each integration. The dispatcher is non-blocking — slow sinks are dropped after the configured timeout, never holding back the request pipeline.

2.9 — Branding

The public landing page, the reveal page and the email body all pick up the instance name, the three brand colours, the contact URL and the inline logo (PNG or SVG, ≤ 256 KB) you configure here. Changes apply on the next page load — no rebuild, no restart.

2.10 — Security keys (WebAuthn)

Each admin can enrol one or more FIDO2 / WebAuthn authenticators — hardware keys, platform passkeys — as an additional second factor. At the moment of writing, TOTP remains the active second factor at sign-in and WebAuthn substitution kicks in automatically when a key is registered.

2.11 — SMTP relay

SealKeeper does not deliver mail directly — it expects an existing relay.

  • Host and port — typically smtp.example.eu:587 for STARTTLS or :465 for implicit TLS.
  • Authentication — username and password, or an app-specific token for Google Workspace / Microsoft 365.
  • From address — must match an allowed sender on the relay, e.g. seals@example.eu.
  • TLS mode — auto picks STARTTLS when the server advertises it; implicit connects directly over TLS; disable is for local capture only.

In eval mode and in public-demo mode the SMTP layer is replaced by an in-memory capture so a misconfigured relay can't accidentally exfiltrate reveal links during evaluation. The captured queue is browsable at /admin/captured-mail.

3 · The public flow

The public surface is intentionally minimal. Any user can submit their email at https://your-host/ and receive the reveal link by email. There is no form-side feedback that distinguishes "allowed" from "denied" — the public flow always answers with the same 202 acknowledgement (FR-B.13).

3.1 — REST API

curl -X POST https://your-host/api/v1/request \
  -H "Content-Type: application/json" \
  -d '{
    "email": "jane@partner.eu",
    "domain": "partner.eu",
    "subject": "Quarterly access renewal"
  }'

The response is always HTTP 202 { "status": "accepted" }. In eval mode it also carries a debug_reveal_url field so smoke tests can exercise the consume path without parsing the inbox.

4 · Reviewing the audit log

Every state-changing action — token issued, token consumed, admin login, policy edit, integration toggle, demo data reset — emits a hash-chained audit row. The page lists them in reverse chronological order, paginated, with an integrity badge that flips red the moment the chain breaks.

RGPD reminder. The audit log contains personal data (email addresses, timestamps tied to individuals). It is subject to Article 30 record-of-processing requirements. A pre-filled fiche de traitement template lives at docs/rgpd/ in the repository.

5 · Backup and disaster recovery

The persistent volume contains:

  • The admins table (bcrypt-hashed passwords, AES-256-GCM-wrapped TOTP secrets).
  • The audit log (SQLite by default, Postgres for production).
  • Branding overrides + logo blob (inline, ≤ 256 KB).
  • Admin-uploaded libraries (content-addressed on disk).
  • Email template overrides and outbound integrations metadata.

For backup, snapshot the volume nightly using your existing tooling. The disaster-recovery procedure is documented in docs/disaster-recovery.md.

6 · Upgrading

SealKeeper follows semantic versioning. Patch releases (0.1.x) are drop-in compatible and require only docker pull. Minor releases (0.x.0) may add new admin settings — review the changelog before upgrading. Major releases (x.0.0) may require migration steps which are documented per release.
