Operating SealKeeper

From the initial Docker container to the daily generation of one-time links, this page walks you through the operational lifecycle of a SealKeeper instance.

1 · Installation

Prerequisites

  • A Linux host with Docker 24+ or Podman 4+, exposed on port 443 (or behind a reverse proxy)
  • A reachable SMTP relay (host, port, username, password) for sending recipient emails
  • A persistent volume of at least 1 GB for ciphertext storage and audit logs
  • A reverse proxy with valid TLS (nginx, Caddy, Traefik, or Cloudflare Tunnel)

Pulling and running the container

# SealKeeper runs as a single Go binary in an OCI image
docker run -d \
  --name sealkeeper \
  --restart unless-stopped \
  -p 8443:8443 \
  -e SK_BASE_URL="https://seal.example.eu" \
  -e SK_ADMIN_INITIAL_PASSWORD="change-this-on-first-login" \
  -v sealkeeper-data:/var/lib/sealkeeper \
  ghcr.io/sched75/sealkeeper:latest
Note. The SK_ADMIN_INITIAL_PASSWORD environment variable seeds the admin console on first launch. You will be prompted to change it immediately upon first login and the seed value will be discarded from memory.

2 · Admin console setup

Navigate to https://your-host/admin in your browser. Log in with the username admin and the initial password set via environment variable. You will be prompted to set a new password and to configure the three core areas: generation policy, SMTP relay, and audit retention.

2.1 — Generation policy

The policy controls what passwords look like and how long they live before purging. The console offers three modes:

ModeDescriptionSuitable for
alphanumericLetters A-Z, a-z, digits 0-9. Configurable length 12-64.Most applications, broad compatibility
symbolicAlphanumeric plus symbols !@#$%^&*. Configurable length 12-64.Systems requiring stronger character variety
passphraseDiceware-style words separated by hyphens. Configurable count 4-8.Human-readable credentials, recovery codes

2.2 — SMTP relay

SealKeeper uses your existing SMTP infrastructure — it does not deliver email directly. Configure:

  • Host and port — typically smtp.example.eu:587 for STARTTLS or :465 for implicit TLS
  • Authentication — username and password (or app-specific token for Google Workspace, Microsoft 365)
  • From address — must match an allowed sender on the relay, e.g. seals@example.eu
  • Reply-to — typically your help desk or the requester's address (configurable per-link)
Test before saving. The admin console includes a Send test email button that delivers a sample message to an address you specify. Use it to validate authentication and DKIM/DMARC before relying on the relay for production.

2.3 — Audit retention

Audit logs retain metadata for each generated link (requester, recipient, timestamp, reason, expiration). They never contain the password or the decryption key. Configure retention period (default: 12 months, max: 84 months — covering legal requirements in most EU jurisdictions).

3 · Generating links

3.1 — Via the generation page (UI)

Any user with access to https://your-host/ can mint a link. The page requires:

  • Recipient email — single valid address (multi-recipient is a future feature)
  • Reason — free-text field, stored in the audit log (e.g. "Onboarding new contractor")
  • Policy override (optional) — pick a different policy than the default for this generation
  • Lifetime (optional) — override the default TTL (default: 30 minutes)

After submitting, the requester sees a confirmation receipt — but never the password itself. The recipient receives the email and is the only party to see the password, once.

3.2 — Via the REST API

# obtain an API token from the admin console: Settings → API tokens
curl -X POST https://your-host/api/v1/links \
  -H "Authorization: Bearer $SK_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "recipient": "jane@partner.eu",
    "reason": "Quarterly access renewal",
    "policy": "strong-24",
    "ttl_minutes": 30
  }'

The API responds with the link identifier and audit ID, but never the password or the URL fragment containing the key — those are only delivered via the recipient's email.

4 · Reviewing the audit log

The admin console offers a searchable log of every generation, with filters by date, requester, recipient, and status (delivered, opened, expired, purged). Export to CSV is available for DPO reviews.

RGPD reminder. The audit log contains personal data (email addresses, timestamps tied to individuals). It is subject to Article 30 record-of-processing requirements. SealKeeper provides a pre-filled fiche de traitement template in docs/rgpd/ on the GitHub repository for adaptation to your organisation.

5 · Backup and disaster recovery

The persistent volume /var/lib/sealkeeper contains:

  • The encrypted admin password hash (Argon2id)
  • The audit log (SQLite by default, Postgres optional)
  • The active ciphertexts (Redis-compatible, ephemeral by design)

For backup, snapshot the volume nightly using your existing backup tooling. For DR, see the disaster recovery procedure in the repository.

6 · Upgrading

SealKeeper follows semantic versioning. Patch releases (0.1.x) are drop-in compatible and require only docker pull. Minor releases (0.x.0) may add new admin settings — review the changelog before upgrading. Major releases (x.0.0) may require migration steps which are documented per release.
