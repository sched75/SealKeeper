Changelog

SealKeeper follows Semantic Versioning. Notable changes between releases are documented here, with security-relevant items tagged explicitly.

Note. SealKeeper is currently in pre-release. No tagged release is available yet. This page lists what is planned for the first versions and acts as a roadmap. Once the first release is tagged, all past versions will appear here.
0.1.0 planned · 2026 Q3 pre-alpha

Initial pre-alpha release. Single-tenant Docker image, manual admin configuration, no automated migrations. Intended for early adopters willing to file issues against a young codebase.

Added

  • Single-binary Go service with embedded admin console at /admin
  • Public generation page at / with recipient and reason fields
  • Three generation policies: alphanumeric, symbolic, passphrase (Diceware EFF wordlist)
  • AES-256-GCM ciphertext storage in Redis with TTL
  • SMTP delivery with STARTTLS and implicit TLS support
  • Audit log signed with HMAC-SHA-256 chained per entry
  • REST API for scripted link generation with bearer tokens
  • Argon2id admin password hashing
  • Health check endpoint at /healthz
  • French and English UI translations

Security

  • Decryption keys generated server-side via crypto/rand and discarded from memory after email composition
  • URL fragment delivery of keys, never transmitted in HTTP requests by compliant browsers
  • Constant-time comparison for admin password verification
  • Rate limiting on admin login (10 attempts per 15 minutes per IP)
  • Content Security Policy headers on all responses
  • HSTS preload-ready response headers

Known limitations at this release

  • Single-tenant only; no organisation or team scoping
  • No TOTP second factor on admin console (planned for 0.2.0)
  • No SAML or OIDC integration (planned for 0.3.0)
  • No mobile-optimised recipient view (planned for 0.2.0)
  • No reproducible build attestation (planned for 0.3.0)

Subject to revision. Open-source roadmaps are written in pencil. Priorities shift with contributor availability, real-world feedback, and security findings. The dates above are intentions, not commitments.

Versioning policy

  • Patch releases (0.x.y) contain only bug fixes and security patches. Always backwards-compatible. Drop-in upgrade.
  • Minor releases (0.x.0) may add new admin settings, new API endpoints, or new generation policies. Backwards-compatible at the configuration level but may require reviewing the changelog before upgrading.
  • Major releases (x.0.0) may introduce breaking changes. Migration steps are documented per release.

Before reaching 1.0.0, breaking changes may also appear in minor releases. The codebase is too young to commit to API stability — that commitment arrives with 1.0.0 itself.
