Threat Model

A document is honest when it states what it cannot do. This page enumerates the attackers SealKeeper considers, the assets it protects, the boundaries of trust it draws, and — most importantly — the threats it does not defend against.

1 · Assets we protect

SealKeeper exists to protect three concrete assets:

AssetSensitivityDefence priority
Generated password plaintextMaximum — the secret itselfCannot persist anywhere except briefly in server RAM
Decryption key KMaximum — yields the passwordTravels only via URL fragment, never logged
Audit metadata (requester, recipient, timestamp, reason)RGPD personal dataStored encrypted at rest, configurable retention

4 · What SealKeeper defends against

  • Passwords in plain-text email. SealKeeper never sends the password through email; only the URL with the decryption key as a fragment, which is opaque in transit.
  • Permanent retention. Ciphertexts purge on first read or TTL expiry. Audit logs retain only metadata, configurable in length.
  • Server-side disclosure. A compromised database yields only ciphertexts whose decryption keys were never persisted — historical seals cannot be retroactively decrypted.
  • Replay of consumed seals. A second visitor to a consumed link receives 410 Gone; the URL cannot be reused.
  • Unauthorised admin configuration changes. The admin console enforces password authentication, brute-force rate-limiting, and optionally TOTP second factor.
  • Tampering with audit logs. Each audit entry is signed with an HMAC chained to the previous entry, making silent deletion or alteration detectable.

5 · What SealKeeper does NOT defend against

Honesty matters more than marketing. Every security tool has limits. The ones below are SealKeeper's. Understand them before deploying — they may or may not be acceptable for your use case.
  • Server compromise during the email window. Between password generation and email delivery (typically 100-500 ms), the key K exists in server memory. An attacker with active code execution on the server during that window could extract it. This is a fundamental architectural trade-off — generating side-server is what allows admin-configured policies, but it means we cannot promise true zero-knowledge for the server.
  • Compromised SMTP relay. The relay sees the email subject, recipient address, and the URL including the fragment K. If the relay is hostile or compromised, the seal is open to it. Use a relay you operate or trust unconditionally.
  • Recipient inbox compromise. If an attacker has access to the recipient's inbox, they can open the link before the legitimate recipient. SealKeeper has no way to authenticate the human reading the email — only that the link was opened by something with access to the inbox.
  • Browser-side leakage on the recipient device. Browser history, screenshots, password managers offering to save the displayed credential — all can preserve the secret beyond its intended single-read.
  • Quantum decryption of stored ciphertext. AES-256-GCM is currently believed to remain secure against quantum attacks with Grover's algorithm halving its effective strength to 128 bits. However, future cryptanalytic advances could change this. If you store ciphertexts long-term despite our recommendations, factor this in.
  • Side-channel attacks on the server CPU. Spectre, Meltdown, microarchitectural attacks on shared cloud tenancy — these are outside SealKeeper's mitigation scope.
  • Subpoenas and lawful intercept. If your jurisdiction compels disclosure of the server's running memory while it holds K, SealKeeper has no cryptographic counter-measure. Operate the server in a jurisdiction whose legal regime aligns with your threat model.

6 · Recommended deployment mitigations

  • Operate the SMTP relay on the same trust zone as SealKeeper, or use a relay with end-to-end encryption like Posteo or ProtonMail Business
  • Set the default TTL to the shortest practical duration (15-30 minutes typically)
  • Enable TOTP second factor on the admin console
  • Restrict network access to the admin console (/admin) by IP range via reverse proxy or VPN
  • Run the SealKeeper container with read-only filesystem and no capabilities
  • Monitor the audit log for anomalous patterns (high volume from single requester, unusual recipients)
  • Review your DPO record-of-processing annually
