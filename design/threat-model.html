Threat Model

A document is honest when it states what it cannot do. This page enumerates the attackers SealKeeper considers, the assets it protects, the boundaries of trust it draws, and — most importantly — the threats it does not defend against.

1 · The architecture in one paragraph

Before listing threats, the threat model must be anchored in what SealKeeper actually does. The request path is the following. A user submits their email to POST /api/v1/request. The server checks the domain allowlist, mints a 256-bit opaque single-use token, persists the token row (email + IP-hash + 15-minute TTL), sends an email carrying https://<instance>/reveal/<token>, and answers the requester with a constant HTTP 202 accepted. The recipient clicks the link. The reveal page asks GET /api/v1/policy?token=<t>; the server consumes the token atomically, returns a policy descriptor (generator id + parameters), and the browser generates a handful of password proposals from that descriptor using crypto.getRandomValues(). The server never sees a generated password — they exist in the user's RAM and nowhere else.

Consequence for this document. Earlier versions of this page talked about a "decryption key K" sent in the URL fragment and a server-side "ciphertext store". Neither exists. There is no key to derive a password from, because no password is ever stored. The URL carries a one-time authorisation token, not a cryptographic seed.

2 · Assets we protect

AssetWhere it livesSensitivity
Reveal token Server row (table request_tokens), set HttpOnly. Becomes a path segment of the reveal URL once emailed. High — anyone holding an unconsumed token can fetch the policy and generate the same proposal set.
Generated password proposals The recipient's browser RAM exclusively. Never crosses the network. Maximum — but never seen by the server, so the server cannot leak them.
Admin credentials bcrypt hash + AES-256-GCM-wrapped TOTP secret (cryptobox, key derived from SK_MASTER_SECRET). Maximum — they command the whole instance.
Audit log SQLite / Postgres table audit_log, SHA-256 hash-chained. RGPD personal data (emails, IP hashes, timestamps).
Configuration Allowlist of domains, policies, elevations, libraries, integrations, branding. Integrity matters more than confidentiality: a tampered policy could weaken every future generation.
JavaScript bundle Served at /static/sealkeeper-generation.umd.js from the binary's filesystem. Maximum — a tampered bundle replaces the local RNG with one the attacker can predict. Integrity is enforced by Subresource Integrity (production builds) and signed releases.

4 · Attackers we consider

AttackerCapabilityGoal
External network adversary Passive sniff on TLS, opportunistic MITM, endpoint scan. Capture an unconsumed reveal token; brute-force token guesses.
Compromised SMTP relay Reads outgoing mail server-side. Lift the reveal URL before the recipient opens it.
Compromised recipient inbox Reads emails delivered to the recipient. Consume the token before the legitimate user.
Server-host insider Root or DB access on the SealKeeper host. Exfiltrate the admin password hash, the TOTP secret blob, or alter the JS bundle.
Malicious admin Legitimate admin login. Mint tokens for arbitrary recipients, weaken a policy, disable integrations.
Hostile browser context XSS, rogue extension, screen-share running on the recipient. Read proposals out of the RAM of the reveal page.
Recipient device after-the-fact Local access to the device that opened the reveal page. Recover the password from browser history, screenshots, or a password manager cache.

5 · What SealKeeper defends against

  • Passwords in plain-text email. The outgoing mail carries a one-time URL only. The password itself never enters the mail relay, the SMTP transcript, or any inbox.
  • Server-side password storage. The server has no row, no column, no log line, no metric and no temp-file that ever contains a generated password. The browser is the sole holder.
  • Replay of consumed tokens. Consumption is a single SQL transaction (atomic UPDATE … WHERE consumed_at IS NULL). A second visit returns HTTP 410 Gone, which the reveal page renders as "already used".
  • Token enumeration. Tokens are 256 bits of crypto/rand entropy, URL-safe base64-encoded. Guessing one before it expires (15 min default) requires more entropy than the public internet can shove through the request pipeline.
  • Anti-enumeration on the request side. The public flow answers an identical HTTP 202 accepted regardless of whether the email's domain is on the allowlist (FR-B.13). An attacker cannot use the request form as a directory.
  • Brute-force admin login. bcrypt for the password hash, a constant-time response window of ~300 ms, account lockout after five failures, optional WebAuthn second factor on top of TOTP.
  • Tampering with the audit log. Each row stores entry_hash = SHA-256(prev_hash || canonical(event)). A single deletion or alteration breaks the chain; the integrity banner on /admin/audit turns red on the next page load.
  • Rate abuse of the public flow. A sliding-window limiter (configurable per-email and per-IP, 1/hour each in demo mode) caps how often a given identifier can mint a token.
  • Cross-site request forgery on admin actions. Every state-changing admin endpoint requires a double-submit CSRF token (cookie + form field). The cookies are HttpOnly + SameSite=Strict + Secure outside eval/demo mode.
  • Demo-mode misuse. A public demo is forced into capture-only SMTP, integration creation is refused, and the admin-controlled tables are wiped every 24 h (FR-H.79).

6 · What SealKeeper does NOT defend against

Honesty matters more than marketing. Every security tool has limits. The ones below are SealKeeper's. Read them before deploying — they may or may not be acceptable for your use case.
  • Compromised SMTP relay. The relay sees the reveal URL. A hostile or compromised relay can race the legitimate recipient to consume the token. Use a relay you operate, or one with technical operator-level guarantees.
  • Compromised recipient inbox. If an attacker reads the inbox, they hold a one-time URL that's about to be used legitimately. SealKeeper has no way to distinguish the legitimate human from the attacker. Mitigate at the inbox level (strong mailbox 2FA, alert on auto-forwarding rules).
  • Tampering with the JS bundle. A server-side attacker that can rewrite /static/sealkeeper-generation.umd.js can replace crypto.getRandomValues() with a function whose output is known to them. The release bundle is published with SHA-256 sums and an in-toto attestation, but enforcement of those is a deployment concern (Subresource Integrity on a CDN, signed Docker image at start, etc.). Operators should pin to a known-good build.
  • Browser-side leakage on the recipient device. Browser history records the URL, a password manager extension may try to capture the displayed credential, a screen recorder is invisible to the page. The reveal page sets Cache-Control: no-store and clears the proposals from the DOM when the tab closes, but these are advisory — not a hard guarantee.
  • Malicious admin. An authenticated admin can lower a policy, disable an integration, mint a token for any address on the allowlist. Defence is purely after-the-fact: the audit log records every mutation, the hash chain detects retroactive edits.
  • Compelled disclosure / lawful intercept. A jurisdiction that compels the operator to hand over the server's running memory or the database has access to active tokens and to all audit entries. SealKeeper offers no cryptographic counter-measure here; operate the instance in a jurisdiction whose legal regime aligns with your threat model.
  • Side-channel attacks on shared hardware. Spectre / Meltdown / cross-tenant cache timing on a shared-host deployment are outside SealKeeper's scope. Use bare metal or single-tenant VMs for high-sensitivity instances.
  • Quantum cryptanalysis of the admin password hash. Even if a future attacker eventually broke bcrypt, they would still not recover any user password (none are stored) — only the admin's. The mitigation is the ANSSI B3 floor on admin passwords (FR-M.4): 16 chars × ≥ 3 character classes, ≥ 100 bits.
  • Long-lived token theft. The default 15-minute TTL is tight but not zero. If an attacker captures the reveal URL within that window and reaches the server first, they win. Reducing the TTL or restricting reveal access by IP range is operator policy, not a built-in lock.

7 · Recommended deployment mitigations

  • Operate the SMTP relay on the same trust zone as SealKeeper, or rely on a relay that provides operator-level transparency (Posteo, ProtonMail Business, in-house Postfix).
  • Pin the reveal-page bundle to a known SHA-256; verify the published sealkeeper-generation.umd.js.sig at deploy time.
  • Keep the default token TTL short (15 min). The reveal page is a one-shot interaction — there is no UX cost to a tight window.
  • Enrol TOTP on every admin account and, where possible, add a WebAuthn / FIDO2 second factor on top.
  • Restrict network access to /admin by IP range (reverse proxy, VPN, mTLS).
  • Run the SealKeeper container with a read-only root filesystem and no extraneous Linux capabilities; mount the data volume rw only.
  • Send every audit event to an external SIEM via the built-in integrations (Integrations). A break in the hash chain is a high-severity alert.
  • Review your DPO record-of-processing annually; tighten the audit-log retention window when no legal requirement demands more.
  • For high-sensitivity instances: enable public-demo-mode-style behaviour (capture-only SMTP, aggressive rate-limit) on a staging instance to validate the request grammar before exposing the production endpoint.
