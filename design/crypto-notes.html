Cryptographic Notes

The cryptographic primitives, the key lifecycle, and the architectural decisions that determine SealKeeper's security properties. This page is written for security reviewers, auditors, and contributors familiar with modern symmetric cryptography.

1 · Algorithms in use

No custom cryptography. SealKeeper composes well-understood primitives from the Go standard library. There is no custom encryption scheme, no rolled-our-own MAC, no novel KDF. The novelty is in the operational envelope around these primitives, not in the primitives themselves.

3 · Password generation specifics

The server reads bytes from crypto/rand.Reader (which wraps getrandom(2) on Linux, BCryptGenRandom on Windows, arc4random_buf(3) on macOS) and maps them to the configured character set using rejection sampling to avoid modulo bias. 

// Pseudocode: bias-free character selection
charset := []byte("ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz0123456789")
threshold := 256 - (256 % len(charset))
for len(password) < targetLength {
  b := make([]byte, 1)
  _, _ = crypto/rand.Read(b)
  if b[0] < byte(threshold) {
    password = append(password, charset[b[0] % byte(len(charset))])
  }
}

The entropy of a generated password is length × log2(charsetSize) bits. For the default alphanumeric policy with 20 characters, that yields ≈ 119 bits — well above the 90 bits considered safe against generic offline cryptanalysis through 2050.

4 · URL fragment design

The decryption key K is encoded as URL-safe base64 (RFC 4648 §5, no padding) and appended to the link after a # separator. The link sent by email therefore looks like: 

https://seal.example.eu/r/01HXR7P9Q3Z8F4N2K6V1J5W2YB#YlpHaG9zYXNzaXNwOXAxbjJzaDM

The choice of URL fragment rather than query string or path is deliberate and rests on RFC 3986 §3.5:

RFC 3986 §3.5. "The fragment identifier component of a URI allows indirect identification of a secondary resource [...] Although the fragment identifier is not used in the scheme-specific processing of a URI, it is appended to the URI for use by the agent. [...] As such, the fragment identifier is not sent to the server."

All major browsers (Chromium, Firefox, Safari, Edge) comply with this specification. When the recipient clicks the link, their browser issues a GET request to https://seal.example.eu/r/01HXR7P9Q3Z8F4N2K6V1J5W2YB without the fragment. The JavaScript on the response page parses window.location.hash locally to obtain K.

5 · Ciphertext storage

Ciphertexts are stored in Redis (or any Redis-protocol-compatible store like KeyDB or Dragonfly) with a TTL set to the policy's configured lifetime. Each entry consists of:

FieldFormatPurpose
id26-char ULIDPublic identifier in the URL path
ciphertextraw bytesAES-256-GCM-sealed password
nonce12 bytesUnique nonce used during encryption
created_atRFC 3339 timestampUsed for TTL and audit
audit_idULIDReference to audit log entry

No password plaintext, no decryption key, no fragment is ever stored. On first successful read, the entry is deleted using DEL in the same Redis transaction as the read, preventing a second visitor from retrieving it even in race conditions.

6 · Audit log integrity

Each audit log entry is signed using HMAC-SHA-256 with a server-side secret loaded at startup from SK_AUDIT_HMAC_KEY. The HMAC is computed over the entry's payload concatenated with the HMAC of the previous entry, forming a hash chain. This makes silent deletion or alteration of past entries detectable on verification. 

entry_N.hmac = HMAC-SHA-256(
  audit_secret,
  entry_N.payload || entry_{N-1}.hmac
)

The admin console offers a Verify audit chain button that walks the chain from the earliest entry and reports any break. This is useful for DPO audits and incident response.

7 · Known limitations

  • Key in server memory during email composition. See the threat model for context. This is the principal architectural compromise.
  • No deterministic build reproducibility yet. The released binaries are built on GitHub Actions with reproducibility flags but not yet attested. Planned for v0.3.0.
  • No post-quantum hybrid encryption. AES-256-GCM is currently sufficient against known quantum attacks; we have no hybrid KEM in place yet. The TTL nature of stored ciphertexts substantially limits exposure, but long-term retrospective decryption by a future cryptanalytic break cannot be ruled out for any seal that lived even briefly in our store.
  • No formal verification. The crypto code has not been mechanically verified with a tool like Cryptol or F*. We rely on the audited primitives of the Go standard library and external code review.
  • No independent third-party audit yet. The codebase is open for audit but has not undergone a paid commissioned review. This is a known item on the roadmap.
