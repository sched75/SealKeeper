User guide

Receiving and using a SealKeeper link

Someone has asked SealKeeper to send you a password — or you have just requested one for yourself. This page walks through what arrives in your inbox, how to consult it, and what becomes of the link afterwards.

1 · The anatomy of a SealKeeper email

A SealKeeper message is short. It identifies the instance that sent it, shows a single call-to-action button, and states the validity window in plain words. There is no attachment, no tracking pixel, and no embedded password.

  • From — the address belongs to the SealKeeper instance, not to your IT department's general mailbox. The display name will say something like SealKeeper · example.eu.
  • Subject — Your SealKeeper password proposals, translated into the language your browser was set to when the request was made.
  • Body — a brief explanation, the validity window in minutes, and a single See my proposals button. The button points at https://<instance>/reveal/<token>.
Doubt the message? Hover the button and read the link before you click. The hostname must be the one your organisation has announced. If anything else appears — a third-party shortener, a look-alike domain, an unfamiliar TLD — treat the email as a phishing attempt and forward it to your IT security desk.

2 · Opening the link

Click the button or paste the URL into your browser. SealKeeper presents a single page with a discrete Decode action. The password is not shown until you press it — that gives you time to check your surroundings (screen visible from the corridor, screen-sharing still running, a colleague looking over your shoulder).

Once you press Decode, the page reveals a small number of candidate passwords — typically five. They all satisfy the same generation policy; pick the one that you find easiest to remember or to type. The choice has no security implication: each proposal carries the same level of strength.

2.1 — Why several proposals?

A single password chosen at random is often awkward to live with — too many ambiguous characters, a syllable that is hard to pronounce, a digit cluster that the eye loses. By offering several candidates, SealKeeper lets you pick a password that fits your fingers and your memory without weakening the cryptography behind it.

2.2 — The countdown

At the top of the reveal page a small timer ticks down. When it reaches zero, the page closes itself, the proposals disappear, and the link cannot be opened again. The default lifetime is fifteen minutes from the moment the email left the relay.

Copy first, then close. Save the proposal you have chosen to your password manager before you close the tab. There is no show again action — SealKeeper is built so that the page cannot be re-opened, by you or by anyone else.

3 · Requesting a password for yourself

Some SealKeeper instances expose a public request page. If your organisation has one, navigate to its base address — for example https://seal.example.eu — and fill in the form:

  • Your email address — the one that will receive the link. Use a mailbox that you control alone; shared mailboxes defeat the one-recipient design.
  • Submit — the page acknowledges the request with a polite confirmation but never tells you whether the email was actually accepted. That silence is intentional: SealKeeper refuses to reveal whether a domain is on the allowlist, so that an attacker cannot use the request form as a directory.

If your email address is allowed by the instance, the link arrives within a few seconds. If it is not, nothing happens — no error, no bounce, no hint. Ask your administrator if you believe you should be allowed.

One request per hour. The public form throttles requests per email and per IP. A second submission within the same hour is silently dropped — again, by design. Wait, or ask an operator to raise the limit for your domain.

4 · What happens after you have used the link

The token in the URL is consumed the moment the proposals are rendered. Re-opening the same link — whether by you, by a colleague who saw the email, or by any program that follows links automatically — yields a short message saying that the link has already been used. The proposals themselves are never written to disk and cannot be recovered.

If your administrator has enabled the optional post-consultation notification, you may receive a short follow-up email recording the exact moment the link was consulted. The mail contains no password and no usable token; it is intended purely as an audit trail for you, the recipient.

5 · Common questions

5.1 — The page says the link has expired

The validity window — fifteen minutes by default — has elapsed. Return to the request page and ask for a new link. The old one is now inert and can be deleted from your inbox.

5.2 — The page says the link has already been used

The token has been consumed. If it was not you, treat the inbox as potentially compromised: someone else has seen the password. Rotate the credential at the target system immediately and review your mail forwarding rules.

5.3 — I clicked but the page is blank

SealKeeper relies on a small JavaScript bundle to decode and display the proposals. If JavaScript is disabled in your browser, the reveal page cannot work. Re-enable it for the SealKeeper hostname only, or open the link in a browser where scripts are allowed.

5.4 — Is the password in the URL?

No. The URL contains only an opaque, random token. The token is the permission to request the password from the server — and permission is granted exactly once. The password itself never appears in a URL, in a query string, in a server log, or in a referrer header.

5.5 — Could someone intercept the link between SealKeeper and my inbox?

SealKeeper expects to be deployed behind HTTPS and to send mail through a relay that enforces TLS. The token is single-use and short-lived, so even a momentary capture has a tiny window of usefulness. Your most effective defence remains the basics: a trusted device, a screen-locked laptop, and a mailbox protected by a strong second factor.

5.6 — Can I forward the email to a colleague?

Technically yes — the link is a URL like any other. Operationally no — the moment your colleague opens it, you have lost both the password and the audit clarity of who consulted it. If you need a colleague to receive a password, ask the requester to mint a new link addressed to them.

5.7 — What if my password manager grabbed the URL automatically?

Some browser extensions follow links in your inbox to compute previews or to detect phishing. If a preview engine consumed the token before you saw the email, the page will report already used. Request a new link and consider disabling automatic link preview for your SealKeeper hostname.

6 · What SealKeeper sees and what it does not

SealKeeper is built so that the smallest amount of data possible crosses its threshold. The following table summarises what is recorded and what is not.

ItemRecorded?For how long
Your email addressYes, in the audit logConfigurable by the operator; typically 12 months
The request timestampYesAs above
A coarse hash of your IP / user-agentYes, for rate-limitingCleared with the audit retention window
The proposals themselvesNoGenerated in-browser, never sent to the server
Your final choiceNoThe server cannot tell which proposal you kept
The target system the password is forNoSealKeeper does not ask
RGPD note. Under EU regulation 2016/679 you may ask the operator of the SealKeeper instance for a copy of the audit entries that mention you, or for their deletion. The operator must answer within one month. The contact address is shown at the bottom of every page of the SealKeeper instance you used.

7 · Reporting a problem

If something feels wrong — a phishing-flavoured email that claims to come from SealKeeper, an unexpected reveal page, an audit entry you do not recognise — first contact the operator of the instance you received the link from. They have the audit log and can reconstruct the event chain.

If the issue is a software vulnerability in SealKeeper itself, our responsible disclosure procedure is documented on the security policy page.
